# 参考项目细节（给 AI）

参考源：`Assets/VariableInventorySystem`

## 1. 必看文件顺序

1. `SampleScene.cs`：系统初始化入口
2. `VariableInventoryCore.cs`：拖拽主流程与状态机
3. `StandardCore.cs`：多视图注册与容器弹窗协作
4. `StandardStashViewData.cs`：网格占位/插入判断
5. `StandardStashView.cs`：UI 落点吸附与预览
6. `ItemCellData.cs`、`CaseCellData.cs`：物品与容器数据

## 2. 参考项目核心链路

- 启动：构建主背包数据 -> 渲染网格与物品
- 拖拽开始：从视图取出物品 -> 显示拖拽影像
- 拖拽中：计算吸附落点 -> 标记合法/非法
- 旋转：交换宽高 -> 重新计算占位
- 拖拽结束：尝试放置 -> 成功写回 / 失败回滚
- 容器点击：打开子背包视图 -> 共享同一 Core

## 3. 复刻时只抄思路

要保留：

1. 多格占位模型（锚点 + 尺寸/旋转）
2. 统一拖拽核心（跨视图）
3. 容器作为“物品持有子背包”

不要照搬：

1. 视图里写业务判定
2. 单个方法里混判定、执行、回滚
3. 过重的 UI 与数据耦合

## 4. 给 AI 的实现约束

- 先保证“可运行闭环”，再加扩展功能。
- 新逻辑优先加到 `GridPlacementService` / `InventoryViewModel`。
- View 脚本只做显示和事件转发，不写放置规则。
- 严格参考 `Assets/VariableInventorySystem` 的运行效果，不照抄实现细节。
- 代码风格以当前 `Assets/GridInventory` 为准（命名、目录、职责边界）。
- 目标是“实现同等体验”，不是“复刻同名类与同结构”。

## 5. Samples 导向约束（必须遵守）

`Assets/GridInventory/Samples` 将承载多个演示场景，因此 Runtime 模块必须满足：

1. 可插拔：任一功能模块（旋转、堆叠、容器、保存）可单独接入。
2. 可禁用：禁用模块后系统仍可运行最小背包闭环。
3. 低耦合：`View` 通过 `ViewModel` 调用能力，不直接依赖具体实现细节。
4. 可组合：不同 Sample 可自由组合功能开关，不要求统一大场景。
